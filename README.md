# HSS Hub 팀 컨벤션

## 🎨 스타일 컨벤션 (Prettier 설정)

---

**Prettier 설정 파일 (`.prettierrc`)**

```json
{
  "tabWidth": 2,
  "semi": true,
  "singleQuote": true,
  "trailingComma": "all"
}
```

- **`tabWidth: 2`**: 들여쓰기 시 **공백 2칸**을 사용합니다.
- **`semi: true`**: 문장의 끝에 **세미콜론(`;`)**을 붙입니다.
- `singleQuote: true`: 문자열에 **작은따옴표(`'`)**를 사용합니다. (권장되는 일반적인 설정)
- `trailingComma: "all"`: 객체나 배열의 마지막 요소 뒤에 후행 쉼표를 붙입니다. (권장되는 일반적인 설정)

## 🏷️ 네이밍 컨벤션

| **대상**                               | **표기법**                                      | **규칙/예시**                                                                  |
| -------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------ |
| **변수**                               | **`camelCase`** (카멜 케이스)                   | - 변수: `userName`, `isLoggedIn`                                               |
| **함수**                               | **`camelCase`** (카멜 케이스)                   | - 함수: **동사 + 명사** 형태 권장 (예: `getUserData`, `handleClick`)           |
| **상수**                               | **`UPPER_SNAKE_CASE`** (대문자 스네이크 케이스) | - `MAX_COUNT`, `API_KEY`                                                       |
| **컴포넌트**                           | **`PascalCase`** (파스칼 케이스)                | - **대문자로 시작** (예: `Header`, `UserProfileCard`)                          |
| **Hook**                               | **`camelCase`** (카멜 케이스)                   | - **`use`** 접두사 사용 (예: `useAuth`, `useFetchData`)                        |
| **CSS 클래스**                         | **`kebab-case`** (케밥 케이스)                  |
| 세 단어 이상이면, 분리하여 작성합니다. | - `user-profile`, `button-primary`              |
| **Boolean 변수**                       | **`camelCase`**                                 | - **`is`, `has`, `can`** 접두사 사용 (예: `isActive`, `hasError`, `canSubmit`) |

## 💾 커밋 컨벤션

**`[타입]: [제목]`** 형태로 작성하며, 제목은 **현재 시제, 명령형**으로 간결하게 작성합니다.

| **타입**       | **의미**                                                              | **예시**                                     |
| -------------- | --------------------------------------------------------------------- | -------------------------------------------- |
| **`feat`**     | **새로운 기능** 추가                                                  | `feat: 사용자 로그인 기능 구현`              |
| **`fix`**      | 버그 수정                                                             | `fix: 메인 페이지 렌더링 오류 수정`          |
| **`docs`**     | 문서 변경 (README, 가이드 등)                                         | `docs: README.md에 설치 방법 추가`           |
| **`style`**    | 코드 포맷 변경 (세미콜론, 공백, 포매팅 등, 기능 변경 없음)            | `style: Prettier 설정에 맞게 코드 포맷 변경` |
| **`refactor`** | 코드 리팩토링 (기능 변경 없이 코드 구조 개선)                         | `refactor: 유틸 함수 분리 및 재사용성 개선`  |
| **`chore`**    | 빌드 시스템, 라이브러리 설치 등 기타 변경 사항 (설정, 패키지 관리 등) | `chore: ESLint 및 Prettier 패키지 설치`      |

### 커밋 메시지 구조 예시

```
feat: 사용자 마이페이지 기본 레이아웃 구현

- 마이페이지 레이아웃 컴포넌트 추가 (MyPageLayout.tsx)
- 사용자 정보 섹션, 주문 내역 섹션 구성
- 기본 스타일링 적용 (kebab-case)
```

## 🌿 GitHub Flow 기반 브랜치 전략

`main` 브랜치를 중심으로 모든 작업이 진행됨

| **브랜치 명** | **역할**          | **특징 및 관리**                                                                                                    |
| ------------- | ----------------- | ------------------------------------------------------------------------------------------------------------------- |
| **`main`**    | **프로덕션 배포** | **최종 안정 버전**이며, 항상 배포 가능한 상태여야 합니다. 직접 커밋을 엄격히 금지하며, PR을 통한 병합만 허용합니다. |

## 🔁 개발 사이클 컨벤션

본 워크플로우는 **`main` 브랜치**를 중심으로 모든 작업을 **이슈 기반의 Feature 브랜치**에서 진행하고, PR을 통해 `main`으로 통합하는 4단계 사이클을 따릅니다.

순서 요약:

이슈 생성 → 로컬에서 브랜치 생성 → 작업 → 작업 후

```bash
# 1. main 브랜치로 이동 및 최신 상태 동기화
git branch main

# feat/#1 브랜치에서 실행 시
# 로컬의 feat/#1에서 원격 메인에 있는 내용이 충돌 안나는 이상 합쳐짐

git pull origin main
# ----- 디폴트로 최신화 하는 방법 --------

# 2. 이슈 만들기

git branch feat/#1
git switch feat/#1
여기서 부터 작업

git add .
git commit -m "feat: 작업한 내용"

# -u 옵션은 깃허브에 그 브랜치가 없을 때만 하면됨
git push -u origin feat/#1
# 깃허브에 이미 있으면 아래처럼
git push origin feat/#1

github 페이지에서 Pull Request 작성할 때
본문에 close #1 처럼 고유 번호 꼭 붙이기!
PR넣으면, 박세윤이 검토 후 Merge
```

### Step 1. 📝 이슈 생성 (Issue Creation)

**목표:** 작업 단위 정의 및 브랜치 번호 확보

| **타입 (Type)** | **의미**         | **예시**       |
| --------------- | ---------------- | -------------- |
| **`feat`**      | 신규 기능 개발   | `feat/101`     |
| **`fix`**       | 버그 수정        | `fix/205`      |
| **`refactor`**  | 코드 구조 개선   | `refactor/112` |
| **`chore`**     | 설정/패키지 변경 | `chore/304`    |
| **`hotfix`**    | 긴급 버그 수정   | `hotfix/501`   |

| **항목**         | **컨벤션/작성 내용**                                                                             |
| ---------------- | ------------------------------------------------------------------------------------------------ |
| **제목 (Title)** | **`[타입] 작업 내용 요약`** 형식으로 간결하게 작성합니다. (예: `[FEAT] 사용자 로그인 모달 구현`) |
| **내용 (Body)**  | **'목표', '상세 작업 목록'**만 포함하여 단순화합니다.                                            |
| **담당자/라벨**  | 작업할 본인을 담당자(Assignee)로 지정하고, 라벨(Label)을 부여합니다.                             |

**✅ 이슈 예시**

| **항목**         | **작성 내용**                    |
| ---------------- | -------------------------------- |
| **제목 (Title)** | `[FEAT] 사용자 로그인 모달 구현` |
| **내용 (Body)**  | **## 🎯 목표 (Goal)**            |

사용자가 아이디와 비밀번호로 로그인할 수 있는 모달 UI와 기본 로직을 구현합니다.
**## ✨ 상세 작업 목록**

- [ ] 로그인 버튼 클릭 시 모달 열기/닫기 기능 구현
- [ ] 입력 폼(아이디, 비밀번호) 상태 관리 및 유효성 검사
- [ ] 로그인 API 연동 (POST /auth/login) |
      | **담당자/라벨** | 담당자: `@본인ID`
      라벨: `enhancement`, `feat` |

💡 결과: 이슈가 생성되고 **이슈 번호(예: #16)**가 부여됩니다.

---

### Step 2. 🌳 브랜치 생성 및 분기 (Branching)

**목표:** 이슈 번호에 연결된 작업 브랜치를 `main`에서 생성

| **항목**      | **컨벤션/규칙**                                              |
| ------------- | ------------------------------------------------------------ |
| **분기 지점** | 항상 **`main`** 브랜치에서 분기합니다.                       |
| **네이밍**    | **`[타입]/#[이슈번호]`** 컨벤션을 따릅니다. (예: `feat/#16`) |

[ 이슈에서 생성한 타입 및 이슈번호를 브랜치 이름으로 그대로 작성합니다. ]

**✅ 명령어 예시 (이슈 #16, 신규 기능)**

```bash
# 1. main 브랜치로 이동 및 최신 상태 동기화
git branch main
git pull origin main

# 2. 작업 브랜치 생성 및 이동
git checkout -b feat/#16
```

---

### Step 3. 💾 작업 및 커밋 (Work & Commit)

**목표:** 작업 이력을 컨벤션에 맞춰 기록

**✅ 커밋 예시**

```bash
# 로그인 폼 컴포넌트 구현
git commit -m "feat: LoginModal 컴포넌트 기본 구조 작성"

# 유효성 검사 로직 수정
git commit -m "fix: 아이디 입력 시 최소 길이 유효성 오류 수정"

# API 연동 완료
git commit -m "feat: 로그인 성공/실패 시 API 응답 처리 로직 구현"
```

---

### Step 4. 🔗 Pull Request (PR) 생성 및 병합 (Merge)

**목표:** 코드 리뷰를 통해 작업 브랜치를 `main`로 안전하게 통합

| **항목**      | **컨벤션/규칙**                                                               |
| ------------- | ----------------------------------------------------------------------------- |
| **푸시**      | 로컬 작업 완료 후 원격 저장소에 푸시합니다.                                   |
| **PR 제목**   | **`[타입](#[이슈번호]): 최종 작업 내용`** 형식으로 작성합니다.                |
| **병합 대상** | **`feat/#16`** → **`main`** 브랜치로 요청합니다.                              |
| **종료**      | PR 본문에 `close #16`을 포함하여, 병합 시 해당 이슈가 자동으로 닫히게 합니다. |

**✅ 명령어 및 PR 제목 예시**

```bash
# 원격 저장소에 푸시
git push origin feat/#16
```

✅ **PR 제목:** `feat(#16): 사용자 로그인 모달 기능 구현 완료`

> 💡 사이클 종료: PR이 승인되어 main 브랜치에 병합되면, 해당 코드는 배포(Deployment)를 위한 준비가 완료된 것으로 간주됩니다. feat/#16 브랜치는 삭제하고 해당 이슈(#16)는 Closed 상태가 됩니다.
